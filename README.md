# 🛡 SENTINEL-ASM

*"바이브코딩이 만든 속도, 그 뒤에 쌓이는 보안 부채를 자동으로 점검하고 고친다."*

## 서론 — 코드는 빨라졌고, 보안은 무너지고 있다

생성형 AI와 **바이브코딩(vibe coding)** 의 등장으로 누구나 며칠 만에 서비스를 띄운다.
그러나 빠르게 생성된 코드일수록 **노출된 `.env`, 방치된 관리자 페이지, 누락된 보안 헤더,
구버전 의존성, 잘못된 CORS** 같은 취약점을 그대로 품은 채 배포된다.
**개발 속도는 10배 빨라졌지만, 보안 점검 속도는 그대로다.** 이 격차가 곧 사고가 된다.

실제로 AI가 생성한 코드의 상당수가 알려진 취약점 패턴을 포함하며,
비전문가가 만든 서비스는 "남들이 잘 안 보는 한 줄짜리 설정 오류" 하나로 데이터가 통째로 새어 나간다.
**SENTINEL-ASM 은 바로 이 지점 — 폭증하는 코드와 정체된 보안 사이의 간극 — 을 메우기 위해 만들어졌다.**

## 기(起) — 격화되는 보안 이슈

| 현상 | 결과 |
|---|---|
| 바이브코딩으로 양산되는 서비스 | 점검되지 않은 공격표면 폭증 |
| 비밀키·환경설정 파일 노출(`.env`, `id_rsa`, 백업) | 인증정보 유출 → 시스템 장악 |
| 구버전 라이브러리(CVE) 방치 | 공개 익스플로잇으로 즉시 침해 |
| 보안 헤더·CORS·쿠키 설정 누락 | XSS·세션 탈취·클릭재킹 |
| **무권한 스캐너의 오남용** | 점검 자체가 불법(정보통신망법)이 되는 역설 |

기존 도구는 "스캔은 잘 하지만" ① 권한 검증이 사람의 약속 수준이고, ② 리포트는 사람이 읽는 PDF에 그쳐
**조치까지 이어지지 않으며**, ③ 흔한 항목만 보고 *남들이 안 보는 디테일*을 놓친다.

## 승(承) — 해법: 권한 검증 기반 보안관제 · 보안 에이전트

SENTINEL-ASM 은 **"검증된 권한이 있는 대상만, 비파괴로, 토시 하나까지 점검하고, 그 결과를 AI가 직접 고치게 하는"** 자율 보안 에이전트다.

- 🔐 **소유권 검증 게이트** — 소유권·동의·범위·기간을 시스템이 검증하지 못하면 *능동 스캔이 기술적으로 차단*된다. (제품의 법적 방어선)
- 🤖 **AI 오케스트레이션** — 발견 → 위험 산정 → 조치 명령 생성 → AI 수정 → **재점검 폐루프**까지 자율 수행.
- 🔬 **간단 / 심층 2단 점검** — 일상 모니터링(간단)과 *남들이 안 보는 것까지 전수*(심층)를 분리.
- 📊 **컴플라이언스 자동 증빙** — 모든 발견을 CWE·OWASP Top 10·ISMS-P·ISO 27001·PCI-DSS·NIST CSF·개인정보보호법/GDPR 에 매핑.

## 전(轉) — 어떻게 동작하는가

```
[권한 검증 게이트] → [비파괴 점검 오케스트레이션] → [위험 산정·키워드 가중]
        ↓ (미검증 시 차단·기록)           ↓
   [감사 해시체인]              [AI 수정 명령서 생성] → [AI 에이전트 조치] → [재점검 delta]
```

1. **검증 게이트**: 소유권 + 동의 + 범위 + 점검 윈도우를 모두 충족해야 큐 진입. 통과 시 **egress allowlist** 를 워커에 주입해 범위 밖 발신을 하드 차단, 동의 철회 시 **kill-switch**.
2. **점검 엔진(비파괴)**:
   - **ASM** — 서브도메인 열거, 오픈 포트(확장 47종), TLS 버전/인증서, **이메일/DNS 보안(SPF·DMARC·DKIM·CAA)**, 서브도메인 탈취
   - **구성** — 보안 헤더 전체(HSTS·CSP·COOP/COEP·Permissions-Policy), **CORS 오구성**, 위험 HTTP 메서드, 쿠키 `__Host-` 규칙, **정밀 민감경로(.env·.git·Actuator·id_rsa·web.config — 콘텐츠 검증 + soft-404 오탐 제거)**
   - **CVE/SBOM** — npm·PyPI·Maven 구성요소 대조(EPSS/KEV 가중)
   - **DAST** — OWASP Top10 비파괴 지표, 오픈 리다이렉트·Host 헤더 반사
3. **위험 산정**: CVSS × 자산중요도 × 노출도 + EPSS/KEV + **고위험 신호 키워드 가중(RCE·비밀키·인증우회 등 ×1.12~1.50)** → "묻히는 한 건"을 상위로.
4. **AI 수정 명령서**: 발견을 *[무엇이 노출 → 왜 위험 → 공격 시나리오 → 비즈니스 피해 → 요구 수정 → 완료 기준]* 구조의 마크다운으로 변환 → Cursor 등 AI 코딩 어시스턴트에 그대로 주입하면 AI가 코드를 수정.
5. **폐루프 재점검**: 조치 후 재점검 → 직전 대비 delta 로 해소 자동 판정.
6. **감사추적**: 모든 행위를 변조 불가(append-only) **해시체인**으로 기록.

방화벽 뒤에서 사람이 로그를 들여다보던 시대는 저문다.

SENTINEL-ASM 은 그 미래의 **보안관제 에이전트** 레퍼런스다 — *권한 안에서만, 빠짐없이, 그리고 끝까지 고친다.*

## ⚙️ 빠른 시작

```bash
npm install
npm run seed     # 데모 테넌트·RBAC 사용자·자산 시드
npm run dev      # http://localhost:8787
```

| 계정 | 비밀번호 | 역할 |
|---|---|---|
| admin@hanbit.example | `sentinel!admin` | 관리자(전체) |
| scanner@hanbit.example | `sentinel!scan` | 점검자 |
| auditor@hanbit.example | `sentinel!audit` | 감사자 |
| viewer@hanbit.example | `sentinel!view` | 조회자 |

> 메인 화면에서 도메인 입력 + 권한 확인 체크 → **간단/심층** 선택 → 즉시 점검.
> 소프트웨어 파일(package.json 등)은 권한 절차 없이 업로드만 하면 CVE 분석.

## ✨ 주요 기능

- 소유권 검증 게이트 · egress allowlist · kill-switch

- 멀티 자산 일괄 점검 · 위험도 추세 대시보드
- 멀티테넌시 · RBAC · 변조방지 감사 해시체인
