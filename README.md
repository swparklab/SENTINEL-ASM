# SENTINEL-ASM

**권한 검증 기반 보안관제·자율 보안 점검 플랫폼**
공격표면관리(ASM) · 취약점 점검(VAPT) · 컴플라이언스 매핑 · AI 기반 조치 연계

본 문서는 SENTINEL-ASM 의 설계 배경, 동작 원리, 실제 적용 방식, 그리고 현재의 한계와
향후 개선 방향을 정리한 기술 개요서이다.

---

## 1. 개발 배경

최근 생성형 AI 와 이른바 *바이브코딩(vibe coding)* 의 확산으로 소프트웨어의 생산 속도는
비약적으로 증가하였다. 그러나 코드를 생성하고 배포하는 속도가 빨라진 것에 비해, 그 산출물에 대한
보안 점검은 여전히 전문 인력의 수작업과 사후 대응에 의존하고 있다. 그 결과 점검되지 않은 공격
표면이 누적되고, 노출된 환경설정 파일, 방치된 관리 인터페이스, 구버전 의존성, 잘못 구성된
보안 헤더와 같은 취약점이 그대로 운영 환경에 반영되는 사례가 늘고 있다.

문제는 두 방향에서 동시에 발생한다. 한편으로는 위와 같이 *점검의 공백* 이 커지고 있으며,
다른 한편으로는 자동화된 취약점 스캐너를 권한 없는 대상에 사용하는 행위가 그 자체로 법적
위험(정보통신망법 등)을 수반한다는 점이다. 즉 "점검을 해야 하지만, 함부로 점검해서는 안 되는"
긴장 관계가 존재한다.

SENTINEL-ASM 은 이 두 문제를 동시에 해결하기 위해 설계되었다. 핵심 전제는 다음과 같다.
**점검 대상에 대한 권한이 시스템 차원에서 검증된 경우에 한하여, 비파괴적인 방식으로, 가능한 한
빠짐없이 점검하고, 그 결과를 사람이 읽는 보고서를 넘어 실제 조치로 연결한다.**

---

## 2. 설계 원칙

| 원칙 | 내용 |
|---|---|
| 권한 검증 우선 | 소유권·동의·범위·기간이 검증되지 않은 대상에 대한 능동 점검은 시스템이 기술적으로 차단한다. |
| 비파괴(non-destructive) | 모든 점검은 익스플로잇 실행이 아니라 취약점의 *존재 여부* 만 식별한다. |
| 결과의 실효성 | 발견 사항을 우선순위화하고, AI 코딩 도구가 즉시 실행할 수 있는 조치 명령으로 변환한다. |
| 표준 정합 | 모든 발견을 CWE·OWASP Top 10 및 주요 컴플라이언스 통제에 매핑한다. |
| 감사 가능성 | 모든 판정·실행·조회를 변조 불가능한 형태로 기록한다. |

## 3. 시스템 개요

시스템은 여섯 개의 논리 계층으로 구성된다.

| 계층 | 구현 위치 | 역할 |
|---|---|---|
| 프레젠테이션 | `public/`, `web/` | 웹 콘솔(React 번들) |
| API 게이트웨이 | `src/auth/`, `src/routes.ts` | 인증(JWT)·인가(RBAC)·멀티테넌시 |
| 오케스트레이션 | `src/modules/orchestrator/` | 작업 큐·동시성·진행률 관리·중단 제어 |
| 검증 게이트 | `src/modules/authorizationGate/` | 소유권·동의·범위·기간 검증 (핵심 모듈) |
| 점검 엔진 | `src/modules/scanners/` | ASM·구성·CVE/SBOM·DAST (비파괴) |
| 데이터 | `src/db/` | 저장소 및 추가 전용(append-only) 감사 해시체인 |

검증 게이트를 통과한 작업에는 발신 허용 목록(egress allowlist)이 주입되며, 점검 워커는 이 목록을
벗어난 대상으로 어떠한 패킷도 발신하지 못한다. 동의가 철회되거나 점검 기간이 만료되면 진행 중인
작업은 즉시 중단된다.

## 4. 실제 적용 방식

### 4.1 운영 워크플로우

1. **대상 등록 및 권한 확인.** 점검 대상(도메인·호스트·소프트웨어 산출물)을 등록하고, 소유권 또는
   점검 권한을 확인한다. 도메인의 경우 입력과 함께 권한 보유 확인 절차를 거치며, 소프트웨어 의존성
   파일(SBOM)은 원격 트래픽을 발생시키지 않으므로 권한 절차 없이 분석한다.
2. **점검 깊이 선택.** *간단(Simple)* 점검은 핵심·고신호 항목을 빠르게 확인하여 일상 모니터링에
   적합하다. *심층(Deep)* 점검은 확장 포트 전수, 정밀 헤더 분석, 공급망·메일/DNS 정밀 점검 등을
   포함하며 시간이 더 소요되는 대신 누락을 최소화한다.
3. **점검 실행.** 오케스트레이터가 모듈별로 점검을 수행하며, 콘솔은 추정값이 아닌 *실제 진행률* 과
   현재 단계를 표시한다. 진행 중 작업은 언제든 중단할 수 있다.
4. **위험 산정 및 표준 매핑.** 발견 사항은 CVSS, 자산 중요도, 노출도에 더해 익스플로잇 확률(EPSS),
   실제 악용 여부(KEV), 그리고 고위험 신호 키워드 가중을 반영하여 우선순위가 산정되고, CWE·OWASP·
   컴플라이언스 통제에 매핑된다.
5. **조치 연계.** 발견 사항은 *무엇이 노출되었는가 → 왜 위험한가 → 공격 시나리오 → 비즈니스 피해 →
   요구 조치 → 완료 기준* 구조의 마크다운 명령서로 변환된다. 이 문서를 AI 코딩 어시스턴트(Cursor 등)에
   전달하면 코드·설정 수정이 자동화된다.
6. **재점검(폐루프).** 조치 적용 후 동일 대상을 재점검하여 직전 결과와의 차분(delta)으로 해소 여부를
   확인한다.
7. **보고 및 증빙.** 경영진 요약, 점검 방법론, 표준 커버리지, 조치 로드맵을 포함한 PDF/Markdown 보고서를
   생성하며, 모든 행위는 감사 해시체인에 기록된다.

## 5. 점검 범위 (현행)

모든 점검은 DNS·HTTP·TLS 수준의 비파괴 관측으로 수행되며, 심층 점검에서 정밀 항목이 활성화된다.

- **공격표면(ASM)** — 서브도메인 열거, 오픈 포트·서비스 식별(심층 47종), TLS 버전·인증서, 서브도메인
  탈취 가능성, IPv6 노출 비대칭, CDN/WAF 식별, 단일 DNS 사업자(SPOF).
- **메일·DNS 정밀** — SPF 과허용 및 DNS 룩업 한도(10회) 초과, DMARC 정렬 모드·서브도메인 정책(sp)·
  집계 리포트(rua), DKIM 셀렉터·테스트 모드·약한 해시, MTA-STS, TLS-RPT, CAA.
- **웹 구성·헤더** — 보안 헤더 전반(HSTS·CSP·COOP/COEP·Permissions-Policy), CSP 정적 분석 및 Trusted
  Types, CORS 오구성(null origin·정규식 결함), 위험 HTTP 메서드, 쿠키 보안 속성 및 프레임워크 지문,
  정밀 민감 경로(콘텐츠 검증 및 soft-404 기반 오탐 제거).
- **API** — GraphQL 인트로스펙션 및 필드 제안 누출, REST 버전 병존, 레이트리밋 헤더.
- **공급망·코드 노출** — 소스맵을 통한 소스 복원, 프론트엔드 번들 하드코딩 시크릿, 서브리소스 무결성
  (SRI) 미적용, 빌드·버전 메타 노출.
- **구성요소 취약점(CVE/SBOM)** — npm·PyPI·Maven 구성요소를 알려진 취약점과 대조하고 EPSS/KEV 가중.

각 항목의 구현 여부와 외부 데이터 연동이 필요한 항목은 [docs/SECURITY-CHECKS.md](docs/SECURITY-CHECKS.md)
의 커버리지 매트릭스에 정리되어 있다.

## 6. 한계 및 개선 방향

본 구현은 데모 및 참조(reference) 목적의 단일 노드 구성이며, 다음과 같은 한계와 개선 과제를 명시한다.

- **외부 인텔리전스 미연동.** 인증서 투명성(CT) 로그, 패시브 DNS, ASN/BGP, 자격증명 유출·다크웹 정보
  등은 외부 유료 API·데이터셋이 필요하여 현재 미구현이다. 향후 `ExternalIntelProvider` 어댑터를 통해
  연동하되, 권한 검증을 통과한 자산에 한해 호출하고 조회 사실을 감사 기록한다.
- **고위험 능동 기법의 분리.** HTTP request smuggling, 캐시 포이즈닝, 타이밍 사이드채널 등은 오탐 및
  서비스 영향 위험이 있어 별도의 승인 환경에서만 수행하도록 분리한다.
- **인증 세션 점검.** 로그인 이후 영역(authenticated surface)의 점검을 위해 세션 쿠키·헤더 주입 방식을
  추가할 예정이다.
- **운영 인프라 치환.** 데이터 계층(현재 파일 기반)과 작업 큐(인메모리)는 운영 환경에서 PostgreSQL,
  메시지 브로커(Kafka 등), 컨테이너 오케스트레이션(Kubernetes), 시크릿 관리(Vault)로 치환하도록
  인터페이스가 설계되어 있다.
- **자동화 확장.** 정기 스케줄 점검, 외부 알림(Webhook/SIEM), 다자산 추세 분석을 추가하여 단발 점검을
  지속적 관제로 확장한다.

## 9. 적법성 및 면책

- 본 시스템의 모든 능동 점검은 사전 권한 검증과 동의를 전제로만 동작하며, 이 전제를 우회하는 사용은
  설계 범위에서 제외된다(정보통신망법 등 관련 법령 준수).
- 권한 없는 대상에 대한 점검은 법으로 금지되며, 시스템이 이를 차단하고 기록한다.
- 본 저장소는 데모·참조 구현으로, 운영 적용 시 별도의 보안 검토가 필요하다.
